# 记忆系统

<p style="font-size: 1.2em;"><strong>是否依赖 CC 沙盒：否</strong></p>

<p style="font-size: 1.2em;"><strong>是否必须绑定工作区：是</strong></p>

记忆系统让 AI 拥有**长期记忆**。

普通对话结束后，AI 不会记住之前聊过什么。启用记忆系统后，工作区中的每次对话和任务都会自动沉淀为记忆。下次交流时，AI 能回忆起之前的讨论、您的偏好、项目的背景——不再需要每次都从头交代。

这意味着工作区用得越久，AI 对您的理解越深，协作效率越高。

![记忆系统总览](/assets/workspace/memory_system/memory_overview.png)

::: tip 零门槛使用
记忆系统不依赖 Claude Code 沙盒。只要有工作区并启用记忆开关，就可以立即开始积累记忆。
:::

## 记忆系统能带来什么

### 不再反复交代背景

「我们项目用的是 FastAPI + PostgreSQL」「上次讨论的方案是 A」——这些您只需要说一次。记忆系统会自动提取并保留这些关键信息。

### AI 对项目的理解持续加深

每次对话都在为 AI 的理解增加深度。一个使用了几周的工作区，AI 对项目结构、技术栈、业务逻辑的理解远超新建的对话。

### CC 沙盒的经验自动沉淀

当 CC 沙盒完成一个复杂任务后，得出的结论和经验会自动沉淀为**语义记忆**，成为长期可复用的知识。例如「项目的认证模块在 /src/auth/，使用 JWT + Redis 方案」。

### 多频道共享记忆

绑定到同一个工作区的多个频道共享记忆。团队成员在不同频道提问，AI 都能基于同一套记忆给出一致的回答。

## 记忆类型

记忆系统采用多层级结构，自动从对话中提取和组织信息：

### 实体（Entity）

从对话中提取的具体对象——人物、项目、技术栈、组织等。实体是记忆系统的基础元素。

> 示例：「用户 Alice」「项目 NekroBot」「数据库 PostgreSQL」

### 关系（Relation）

实体之间的关联。关系记忆带有半衰期，长期不被提及的关系权重会逐渐降低。

> 示例：「Alice 负责 NekroBot 的后端开发」「NekroBot 使用 PostgreSQL 作为主数据库」

### 段落（Paragraph）

记忆系统的核心载体，分为两类：

- **情景记忆**：从日常对话中自动沉淀，记录发生过的事情。例如「5 月 15 日讨论了数据库迁移方案，决定使用 Alembic」
- **语义记忆**：从 CC 沙盒任务结果中沉淀，记录稳定可复用的经验。例如「项目的部署流程是先运行测试，再构建镜像，最后推送到 registry」

### Episode

段落的聚合体。系统自动将相关的段落聚合为 Episode，形成更完整的事件脉络。

## 运行机制

### 自动沉淀

记忆沉淀完全自动，无需手动操作。系统在后台持续监控工作区内的消息，当满足以下条件之一时触发：

- 未沉淀的消息达到阈值（默认 50 条）
- 距上次沉淀超过时间阈值（默认 2 小时）

沉淀过程调用 LLM 从消息中提取实体、关系和事件摘要，并生成向量存储到 Qdrant。

### 智能检索

AI 处理任务时，系统自动执行记忆检索：

1. 将当前对话上下文生成查询向量
2. 在 Qdrant 中检索相似的段落记忆
3. 通过实体和关系进行补充召回
4. 按相关性、时间衰减等因素排序
5. 将召回的记忆注入到 AI 提示词中

### 自然衰减

记忆并非永久不变：

- 关系记忆的半衰期默认为 1 天，长期不被提及的关系会逐渐淡化
- 语义记忆的半衰期更长（默认 30 天），适合稳定的知识
- 后台定期清理低于阈值的记忆（默认每 6 小时执行一次）
- 被反复提及的记忆会持续强化

这种机制模拟了人类记忆的自然规律：重要的信息持续强化，不再相关的信息逐渐淡出。

## 前置条件

启用记忆系统需要：

1. 将 `MEMORY_ENABLE_SYSTEM` 设为 `true`（系统配置 → 基本配置）
2. 配置可用的 embedding 模型组（`MEMORY_EMBEDDING_MODEL_GROUP`）
3. 确认 embedding 维度与模型匹配（`MEMORY_EMBEDDING_DIMENSION`，默认 1024）
4. 配置用于情景沉淀的模型组（`MEMORY_CONSOLIDATION_MODEL_GROUP`）
5. Qdrant 向量数据库正常运行

## 记忆重建

如果记忆数据出现异常，或希望从历史消息中重新提取记忆，可以使用记忆重建。

![记忆重建入口](/assets/workspace/memory_system/memory_rebuild.png)

重建会清除当前工作区的记忆并从历史消息中重新沉淀。默认回溯 30 天（可通过 `MEMORY_REBUILD_LOOKBACK_DAYS` 配置）。

::: warning 重建成本
记忆重建会消耗较多的 LLM 调用和时间。建议在非高峰期执行，且仅在确实需要时使用。
:::

## 关键配置项

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `MEMORY_ENABLE_SYSTEM` | `false` | 记忆系统总开关 |
| `MEMORY_CONTEXT_MAX_LENGTH` | `1200` | 注入提示词的最大字符长度 |
| `MEMORY_CONSOLIDATION_MSG_THRESHOLD` | `50` | 触发自动沉淀的消息数阈值 |
| `MEMORY_CONSOLIDATION_TIME_THRESHOLD_HOURS` | `2.0` | 触发自动沉淀的时间阈值 |
| `MEMORY_RETRIEVAL_DEFAULT_LIMIT` | `10` | 单次检索返回的候选上限 |
| `MEMORY_RETRIEVAL_MIN_SIMILARITY` | `0.5` | 向量检索最低相似度 |
| `MEMORY_PRUNE_ENABLED` | `true` | 是否自动清理低价值记忆 |

完整配置项说明参见 [系统配置](/docs/02_quick_start/config/system#🧠-记忆系统配置)。

## 使用建议

- **尽早开启**：工作区创建后就建议启用记忆系统，越早开始积累，AI 的理解越深
- **保持稳定**：长期工作区的记忆越用越有价值，避免频繁重建
- **描述具体**：与 AI 对话时尽量描述清楚上下文，有助于记忆系统提取更准确的信息
- **与知识库配合**：知识库存放静态资料，记忆系统积累动态上下文，两者互补
- **调优检索**：如果记忆检索效果不佳，可以开启增强检索（`MEMORY_ENABLE_ENHANCED_RETRIEVAL`）
